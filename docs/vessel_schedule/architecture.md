## 1. System Overview

Block 4 adds two read-only API endpoints to the existing `voyage_spine` module and builds two frontend pages on top of the existing React shell.


### Schedule query

Temporal overlap predicate:
```sql
WHERE voyages.commencing_datetime <= :date_to
  AND voyages.expected_completing_datetime >= :date_from
  AND vessels.status = 'Active'
```

Optional filters applied dynamically:
- `vessel_ids`: `vessel_id IN (:ids)`
- `status[]`: `status IN (:statuses)`
- `search`: `voyage_no ILIKE '%:q%'`

Eager loading: `selectinload(Voyage.itinerary_lines)` batches itinerary lines in one additional query. No N+1.

DB indexes added in Block 4 migration (D-LOCK-7):
- `ix_voyages_window (commencing_datetime, expected_completing_datetime)`
- `ix_voyages_vessel_window (vessel_id, commencing_datetime, expected_completing_datetime)`
- `ix_voyages_status (status)`

### Response shapes

`VesselScheduleResponse`:
```
vessels: [
  {
    vessel_id, vessel_name,
    voyages: [
      {
        voyage_id, voyage_no, status,
```


### ECharts integration (D-LOCK-1)

```tsx
// VesselScheduleChart.tsx
const chartRef = useRef<HTMLDivElement>(null)
const instanceRef = useRef<echarts.ECharts | null>(null)

useEffect(() => {
  const chart = echarts.init(chartRef.current, null, { renderer: "canvas" })
  instanceRef.current = chart
  chart.on("click", handleBarClick)
  return () => { chart.off("click"); chart.dispose() }
}, [])

useEffect(() => {
  instanceRef.current?.setOption(buildScheduleOption(data), { notMerge: true })
}, [data])
```

`renderItem` draws each voyage bar as a `rect` + `text` + optional `circle` (alert dot, dormant).

### DOM overlay hit-targets (D-LOCK-4)

A `<div>` positioned absolutely over the ECharts canvas renders one `<button data-testid="voyage-bar-{voyageId}">` per visible voyage. Positioned using the same coordinate math as the chart bars. Playwright targets these buttons.

### Filter state (D-LOCK-6)

```ts
// Route search schema
const scheduleSearchSchema = z.object({
```

TanStack Query key: `["schedule", filters]`. Changing any filter invalidates the cache and refetches.

## 4. Core Flows

### 4.1 Operator opens Vessel Schedule

```
1. Navigate to /schedule
2. TanStack Router reads search params → validated filters (defaults to today ±30 days)
3. TanStack Query fetches GET /api/v1/schedule?date_from=...&date_to=...
4. scheduleChartOption() transforms response → ECharts option
5. VesselScheduleChart useEffect fires setOption()
6. ECharts renders vessel rows + voyage bars on canvas
7. DOM overlay renders voyage bar buttons over canvas
```

### 4.2 Operator clicks a voyage bar

```
1. Button click OR ECharts canvas click fires handleBarClick(voyageId)
2. TanStack Router navigate() to /voyages/{voyageId}/workspace
3. TanStack Query fetches GET /api/v1/voyages/{id}/workspace
4. Workspace page renders header + itinerary table + notes
```

### 4.3 Operator applies a filter

```
1. Filter bar state change → router.navigate({ search: newFilters })
2. URL updates with new search params
3. TanStack Query key changes → refetch schedule endpoint
4. ECharts setOption() with notMerge:true → chart re-renders
```

### 4.4 Back from Workspace to Schedule

```
1. "Back to Schedule" button → router.navigate({ to: "/schedule" })
2. TanStack Router restores search params from history
3. Filter state and chart view restored
```

## 5. Auth

All routes (`/schedule`, `/voyages/$id/workspace`) are inside the `_authenticated` layout route from Block 3 M2. `<RequireAuth>` already guards them. All `openapi-fetch` calls include `credentials: "include"`.
