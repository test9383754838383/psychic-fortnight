# Block 3.5 — Auth + RBAC · Runbook

This runbook outlines operational procedures, common troubleshooting, and development workflows for the **Auth + RBAC** module (`User`, `Role`, `UserRole`, `Session`).

---

## 1. Local Setup & Prerequisites

### One-Command Boot (Backend)
To start the local development FastAPI server:
```bash
make dev
```
This boots the FastAPI application using `uvicorn` on [http://localhost:8000](http://localhost:8000).
- **Database File**: Located at `dev.db` in the project root (SQLite).
- **Default Port**: `8000`.

### Environment Variables (12-Factor)
Auth introduces specific environment variables. In local development, defaults are provided via `src/config.py`. For testing or overriding locally, create a `.env` file in the project root:
```env
SESSION_SECRET=dev-secret-change-me
SESSION_IDLE_TIMEOUT_MINUTES=30
SESSION_ABSOLUTE_TIMEOUT_HOURS=8
SESSION_PURGE_INTERVAL_HOURS=6
LOGIN_RATE_LIMIT=5/minute
COOKIE_SECURE=False  # Set to True in production
ARGON2_TIME_COST=2
ARGON2_MEMORY_COST=19456
ARGON2_PARALLELISM=1
```

## 2. Migrations & Schema Notes

### Initial Seed Data
The Alembic migration for Auth (`add_auth_tables`) automatically seeds the three required roles into the `roles` table:
- `Admin`
- `Operations`
- `Viewer`

### Adding Users (Local/CLI)
Because there is no open registration endpoint, the first `Admin` user must be created programmatically or via a database script if not using the automated tests. In tests, the `AuthService` is used directly. For a quick local user:

```bash
uv run python -c "
import asyncio
from src.dependencies import AsyncSessionLocal
from src.modules.auth.services.auth_service import AuthService

async def create_local_admin():
    async with AsyncSessionLocal() as session:
        auth = AuthService(session)
        await auth.create_user('admin', 'admin', ['Admin'])

asyncio.run(create_local_admin())
"
```

## 3. Backend Auth/RBAC Verification

You can verify the backend endpoints using `curl`. Note that session management relies on HTTP cookies.

### 1. Login
```bash
curl -X POST http://localhost:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "admin"}' \
  -c cookies.txt
```
*Stores the `session_id` cookie in `cookies.txt`.*

### 2. Verify Session (`/me`)
```bash
curl -X GET http://localhost:8000/api/v1/auth/me \
  -b cookies.txt
```

### 3. Verify RBAC (Admin-only Endpoint)
```bash
curl -X GET http://localhost:8000/api/v1/admin/users \
  -b cookies.txt
```

### 4. Logout
```bash
curl -X POST http://localhost:8000/api/v1/auth/logout \
  -b cookies.txt -c cookies.txt
```

## 6. Common Failure Modes and Fixes

### Rate Limit Exceeded (429)
- **Problem**: Repeatedly hitting `/login` fails with `429 Too Many Requests`.
- **Diagnostics**: `slowapi` enforces limits per IP based on `LOGIN_RATE_LIMIT` (default 5/minute).
- **Resolution**: Wait for the rate limit window to expire. In tests, the limiter is disabled if `TESTING=True`.

### Session Not Found or Expired (401)
- **Problem**: API calls return `401 Unauthorized` despite having logged in previously.
- **Diagnostics**: The session exceeded `SESSION_IDLE_TIMEOUT_MINUTES` or `SESSION_ABSOLUTE_TIMEOUT_HOURS`, or the server purged the session row.
- **Resolution**: Prompt the user to re-authenticate.

### MissingGreenlet / sqlalchemy.exc.MissingGreenlet
- **Problem**: Tests or async endpoints fail with a `MissingGreenlet` error during relation access.
- **Diagnostics**: Attempting to access a lazy-loaded relationship (like `User.user_roles`) outside of the initial async database query context.
- **Resolution**: Use `lazy="selectin"` or `lazy="joined"` in the SQLAlchemy relationship definition, or explicitly load the relation using `options(selectinload(...))` during the query.

### Module Boundary Violation (Tach)
- **Problem**: `make tach-check` fails with imports from `src.modules.auth.*`.
- **Diagnostics**: Other modules must not import directly from auth internals.
- **Resolution**: Route dependencies like `get_current_user` and `require_role` through `src/dependencies.py`.

## 7. Session Purge and Scheduler Notes

Background job execution has migrated to `APScheduler` (per ADR-0013). 
- A job named `purge_sessions` is registered during the FastAPI application `lifespan` startup.
- It executes `AuthService.purge_expired_sessions()` every `SESSION_PURGE_INTERVAL_HOURS` (default: 6 hours).
- **Note**: The scheduler runs within the FastAPI event loop. If running multiple web workers (e.g., via Gunicorn with `workers > 1`), multiple scheduler instances will run. Since the purge operation is idempotent and deleting from the database, this is acceptable for V1.

## 8. Security-Sensitive Operational Notes

- **Cookie Security**: `COOKIE_SECURE` is `False` by default for local HTTP development. This **MUST** be set to `True` via environment variables in production to enforce HTTPS-only cookies.
- **Password Hashing**: We use `argon2-cffi`. Production defaults (`t=2`, `m=19456`, `p=1`) meet OWASP minimums. During automated tests, these are dynamically lowered (`t=1`, `m=256`) via the `fast_argon2` fixture in `conftest.py` to keep the test suite fast.
- **No Password Enumeration**: The `/login` endpoint explicitly hashes a dummy password if the username is not found to prevent timing attacks that reveal valid usernames. Always return the same generic `Invalid credentials` error.
- **Admin Password Reset**: V1 lacks a self-service password reset. The `PATCH /api/v1/admin/users/{id}` endpoint allows an Admin to manually override a user's password hash if they are locked out.
